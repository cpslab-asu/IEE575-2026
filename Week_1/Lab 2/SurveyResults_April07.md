## What's Changing Based on This

### 1. The notebook is now step-by-step
The mini project used to dump all tasks at once. Now it's broken into small cells (Steps A–G) so you build the plot one piece at a time.

### 2. Every code block has a plain-English explanation
If "understanding code" is the biggest struggle, we should explain the code *before* you run it — not after.

### 3. "Predict before you run" is the format
Before every code cell, you'll write what you *think* will happen. This is the single most effective way to actually learn what the code does.

### 4. AI is welcome — with one rule
If you ask AI to fix your code or explain something, **write the explanation in your own words as a comment before moving on.** Pasting a fix you don't understand doesn't count as learning.

### 5. Pair up and talk
If you're confident, explaining something to someone else is the best way to deepen your own understanding. If you're struggling, hearing a classmate explain it is often clearer than reading documentation.

### 6. Plotting syntax isn't the goal
We'll provide reference patterns for `plt.plot`, `plt.scatter`, labels, and legends. Don't get stuck on syntax — focus on *what you're trying to show and why*.
